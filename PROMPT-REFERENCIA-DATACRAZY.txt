# DataCrazy Gateway — Referência Técnica Completa

## O que é
Gateway que conecta o CRM DataCrazy ao Meta Conversions API (CAPI).
Quando um lead muda de estágio no CRM, dispara evento (Lead, Purchase, etc) direto pro Meta Pixel via server-side.

## Estrutura do Projeto
```
app/
  main.py              — FastAPI app, lifespan, cron start
  core/
    config.py          — Settings (env vars)
    database.py        — SQLAlchemy async engine
    auth.py            — API key auth (master + client)
  models/
    client.py          — Client model (name, pixel_id, meta_access_token, crm_credentials JSONB)
    event.py           — Event model (client_id, event_type, event_data, user_data, meta_response)
  api/
    clients.py         — CRUD clientes (POST/GET/PATCH/DELETE /api/clients)
    events.py          — Track + webhook + histórico (/api/events)
    crm.py             — Endpoints CRM DataCrazy (/api/crm/pipelines, leads, businesses)
    schemas.py         — Pydantic schemas
  services/
    meta_capi.py       — Envia eventos pro Meta (hash PII, build payload, POST graph API)
    crm_sync.py        — Cron sync: polling CRM a cada 5min, dedup, disparo automático
    datacrazy_service.py — Client HTTP para API DataCrazy
static/
  index.html           — Dashboard completo (SPA vanilla JS)
```

## Como funciona o fluxo
1. Cliente cadastrado com pixel_id + meta_access_token + token CRM DataCrazy
2. No dashboard, configura "Mapeamento de Estágios" — ex: "PASSADO PARA O ESPECIALISTA" → Lead
3. Cron roda a cada 5 min, busca deals que mudaram de stage no CRM
4. Se o stage está mapeado → dispara evento Meta CAPI com dados do lead (email, telefone, nome)
5. Dedup: nunca re-dispara o mesmo deal+stage (checa tabela events antes)
6. Event ID determinístico (sha256) para dedup adicional no Meta (janela 48h)

## REGRAS CRÍTICAS
- DEFAULT_STAGE_MAP está VAZIO — nenhum stage dispara automaticamente
- O cliente DEVE configurar manualmente quais stages disparam quais eventos no dashboard
- Sem stage_map configurado = ZERO eventos disparados (testado e validado)
- Dedup por business_id + event_type na tabela events (nunca re-dispara)
- Token CRM é POR CLIENTE (crm_credentials.datacrazy_token), não global
- Endpoints CRM aceitam ?client_id= para usar o token do cliente correto

## Endpoints Principais

### Clientes
- POST /api/clients — Criar cliente (retorna api_key)
- GET /api/clients — Listar todos
- PATCH /api/clients/{id} — Atualizar (name, pixel_id, events_enabled, crm_credentials)
- DELETE /api/clients/{id} — Remover + eventos

### Eventos
- POST /api/events/track — Disparo manual (controle total)
- POST /api/events/webhook — Webhook genérico (mapeia evento CRM → Meta)
- GET /api/events — Histórico com filtros
- GET /api/events/stats — Métricas (total, by_status, by_type)

### CRM DataCrazy
- GET /api/crm/pipelines?client_id= — Pipelines e stages
- GET /api/crm/leads?client_id= — Leads do CRM
- GET /api/crm/lead-fields?client_id= — Campos disponíveis para field mapping
- GET /api/crm/businesses?client_id= — Negócios/deals
- POST /api/crm/fire-event?client_id=&lead_id=&event_type= — Disparo manual do CRM

### Sistema
- GET /api/health — Health check
- POST /api/sync — Trigger manual do cron sync

## Deploy
O Dockerfile usa git clone (não imagem Docker):
1. git push origin master
2. Atualizar CACHE_BUST no Dockerfile via EasyPanel API
3. Deploy via EasyPanel API
```bash
# Atualizar Dockerfile
curl -X POST "http://95.111.249.160:3000/api/trpc/services.app.updateSourceDockerfile" \
  -H "Authorization: Bearer {EASYPANEL_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"json":{"projectName":"wordpress","serviceName":"datacrazy-api","dockerfile":"...CACHE_BUST=NOVO_HASH..."}}'

# Deploy
curl -X POST "http://95.111.249.160:3000/api/trpc/services.app.deployService" \
  -H "Authorization: Bearer {EASYPANEL_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"json":{"projectName":"wordpress","serviceName":"datacrazy-api"}}'
```
